# 火星文翻译器 V1.0 Sprint 计划与任务分解

> 工程执行视角（非战略路线图）。范围：R01–R07 + R14。团队：Web 小团队 3–5 人（前端 / 设计 / 数据 / 可选后端）。纯规则引擎，无联网 NLP。全程质量护栏贯穿。
> **依赖挂接说明**：`dev-ready-mars-translator-2026-07-06.md`（研发交付说明书，第一部分《研发技术设计说明书》）是引擎/词库/接口的唯一蓝图（SSOT），与本文件同目录。下表任务「依赖」列中的「文档§X」即指该文档的第 X 节：§1 主路径零联网与分享卡策略 / §2 规则引擎与可复现 / §3 数据模型与繁简识别 / §4 配置模型 / §5 接口与存储契约 / §6 边界处理 / §7 质量护栏落地。

## 一、V1.0 目标与 DoD（Definition of Done）

**交付目标**：上线纯规则引擎驱动的汉语↔火星文双向 Web 翻译器，支持双入口、复制、本地历史、收藏、图片分享卡与规则可配置（7 类规则开关 + 强度 + 字符集白名单 + 风格模板），质量三指标达标并通过灰度。

**DoD 判定标准（全部满足方可发布）：**
- **功能完整**：R01–R07、R14 全部实现且可现场演示；边界情况（空输入 / >2000 字分块 / 混合语言路由 / 无法翻译字符原样保留 + 标记）已处理。
- **质量三指标达标**：
  - 词级编辑距离近似准确率 ≥ 规划底线（主指标）；
  - 火→汉反向还原准确率 ≥ 规划底线，且较上版下降 >2pp 即阻断；
  - 确定性映射可复现率 ≥ 99%。
- **回归门禁**：规则更新触发全量回归，词级 / 反向还原准确率下降 >2pp 阻断发布；CI 自动回归基线 **≥2000 条**、月级人工抽检 **≥200 条** 已接入 CI。
- **灰度通过**：0→5%→30%→100% 各阶段退出标准达成，全程未触发回滚条件。
- **发布门槛**：翻译成功率 ≥90% / 可复现率 ≥99% / 分享卡失败率 ≤3%。

## 二、Sprint 拆分（6 周 / 3 个 Sprint）

**Sprint 1（第 1–2 周）· 内核奠基**：脚手架 + 规则引擎核心 + 核心词库 + 汉→火（R01/R03）+ 准确率测试集基线。
**Sprint 2（第 3–4 周）· 双向闭环**：火→汉还原（R02）+ 双入口布局 + 复制（R04）+ 本地历史（R05）+ 边界处理。
**Sprint 3（第 5–6 周）· 体验收口与发布**：收藏（R06）+ 图片分享卡（R07）+ 规则可配置（R14）+ 设置页 + 灰度发布。

## 四、并行轨道

- **质量护栏**：T05（S1，基线 ≥2000 条）→ T11（S2）→ 持续周回归，与研发并行，作为 CI 门禁贯穿 S1–S3；规则更新即全量回归、降 >2pp 阻断；月级人工抽检 ≥200 条。
- **设计**：T17 分享卡视觉规范 S1 即与研发并行定稿（文档§1 安全字符优先+兜底预览）供 T14；T06/T16 双入口与设置页视觉随前端同步走查。
- **数据**：T03 词库与 T05 测试集 S1 与引擎并行，T18 灰度监控 S3 收口。

## 五、发布门槛与灰度交接

**回滚条件（任一触发即回滚并冻结新功能）**：翻译成功率 <90% / 可复现率 <99% / 分享卡失败率 >3%。

**灰度阶段与退出标准：**
- **0→5%（内测）**：内部 + 早鸟邀请码；退出：核心翻译与还原准确率达标、分享卡渲染正常、无 P0 缺陷。
- **5%→30%（分群）**：社交娱乐渠道优先；退出：漏斗（输入→翻译→操作）斜率正向、WSTA 上升、三指标稳定。
- **30%→100%（全量）**：退出：翻译成功率≥90%、可复现率≥99%、分享卡失败率≤3%，且连续 48h 无回滚触发。
- 灰度期每日回流失败样本（乱码/过滤变形）→ 规则热更新 → 词库周更。

## 六、风险与缓冲

1. **确定性映射未达 ≥99% 可复现**：T02 留 1 SP 收敛；确定性流水线优先、禁随机候选、单测锁基线（文档§7 阻断阈值）。
2. **简繁误判**：T04 用内置~3000字双向映射表优先 + Unicode 区间路由（文档§3），OpenCC 默认不引入；测试集增简繁混合用例，误判即补规则。
3. **分享卡变形（微信/QQ 特殊字符过滤）**：T17 安全字符优先 + 兜底预览（文档§1）；失败率>3% 即回滚，降级为纯文本卡。
4. **火→汉还原不唯一 / 降 >2pp 阻断**：T07+T12 回退「原样保留 + 标记」（文档§2/§6）；月抽检 ≥200 条人工校准。
5. **前端资源紧张（3–5 人）**：T14/T16 设计前置并行，R14 留 1 SP 全栈缓冲；复用组件库，必要时降级风格模板数量。
