## 2. 规则引擎设计

处理流水线（逐字符路由 + 词级双向映射）：

1. **字符级语言路由**：按 Unicode 区间逐字判定归属——CJK 统一表意文字（`U+4E00`–`U+9FFF`）及扩展 A 走「汉」分支；平假名（`U+3040`–`U+309F`）/ 片假名（`U+30A0`–`U+30FF`）走「日」分支；注音（`U+3100`–`U+312F`）走「注音」分支；ASCII（`≤U+007F`）走「英/符号」分支；其余原样保留并打 `marks`。
2. **规则族匹配**：对「汉」分支字符，依次查询 7 类规则（同音 / 形近 / 注音 / 拆字 / emoji / 假名 / 拼音首字母），命中即生成候选集。
3. **候选生成**：每类规则产出 1–N 个候选（来自 dictionary 映射表候选集）。
4. **后处理 / 去重**：合并多类候选、去重、应用字符集白名单过滤（③⑤ 类受白名单约束）。
5. **确定性选择**：按「强度 → 优先级 → 候选序」三级确定性策略选唯一输出。

**反向还原（R02）**：dictionary 同时维护反向索引（火星文字符 → 原汉字），`mars→han` 方向走反向索引 + 形近/同音回查，保证可还原。

**可复现保证（可复现率 ≥ 99%）**：引擎全程无随机源；强度映射为「确定性伪随机种子」（`seed = hash(text + JSON(config))`），替换位置 / 数量由种子推导而非 `Math.random`；候选选择固定取候选集最高优先级且稳定排序；所有字典加载为冻结不可变对象。**同输入 + 同配置 ⇒ 同输出**，仅因词库更新导致的语义差异不计入可复现失败。

## 3. 核心数据模型

词条 / 映射表（JSON 示例）：

```json
{
  "id": "tone_雨",
  "han_simple": "雨",
  "han_traditional": "雨",
  "rule_family": "homophone",
  "candidates": ["宇", "语", "羽"],
  "priority": 90,
  "source": "core_dict_v1"
}
```

7 类规则数据表示：

| 规则 | 字段要点 |
|---|---|
| ①同音字替换 | `han_simple/traditional` + `candidates`(同音字) |
| ②形近字替换 | `han` + `candidates`(形近，如 你→伱/妳) |
| ③注音符号混入 | `han` + `bopomofo`(ㄅㄆㄇ) + 插入位置 |
| ④拆字重组 | `han` + `parts`(如 ["木","目"] → 拆解) |
| ⑤emoji/符号穿插 | `han` + `symbols`(["✿","☆"]) |
| ⑥日语假名借用 | `han` + `kana`(きゃりー) |
| ⑦拼音首字母缩写 | `han` + `initials`("wj") |

繁简识别方案：**内置映射表优先**——维护一份 CJK 简↔繁双向映射（约 3000 常用字 + 扩展 A 高频字），O(1) 查表、零依赖、离线可用；OpenCC 类库在 V1.0 作为可选增强挂载（体积 / 准确性权衡），**默认不引入**。建议 V1.0 用内置表覆盖常用字即可。

## 4. 配置模型（R14）

配置对象 schema：

```ts
interface TranslateConfig {
  strength: 'low' | 'mid' | 'high';          // 替换密度
  families: Record<RuleFamily, boolean>;     // 7 类规则开关
  traditional: boolean;                       // 繁体输出开关
  whitelist: { symbols: string[] };           // 符号白名单
  style: 'anime' | 'retro' | 'minimal' | 'custom';
}
```

- **强度映射**：`low`≈替换率 15%（仅高频字、首选候选）；`mid`≈45%；`high`≈80%（多候选 + 多规则族叠加）。强度→替换概率用确定性密度曲线，非随机。
- **字符集白名单**：`whitelist.symbols` 为允许出现的符号 / emoji 集合；命中白名单才允许 ③⑤ 类插入，否则跳过（应对微信 / QQ 过滤）。
- **风格模板 = 规则族预设组合**：`anime` = {homophone, emoji, kana: on; bopomofo, split: off}；`retro` = {homophone, initials: on; kana: off}；`minimal` = {homophone: on; 其余 off}。
- **实时生效**：配置为响应式 store（Zustand / Pinia），变更触发引擎重算，UI 即时预览，无需重载。

## 5. 接口 / 模块契约

核心函数：

```ts
translate(
  text: string,
  srcLang: 'han' | 'mars',
  tgtLang: 'han' | 'mars',
  config: TranslateConfig
): { result: string; candidates: Candidate[][]; marks: Mark[] }
```

- **入参**：`text` 待译文本；`srcLang/tgtLang` 语言方向（汉↔火）；`config` 见 §4。
- **出参**：`result` 最终译文；`candidates` 每字符候选集（调试 / 可解释）；`marks` 保留标记数组（无法翻译字符位置 + 类型）。
- **错误码**：`E_EMPTY`(空输入)、`E_OVERFLOW`(>2000 待分块，由调用方处理)、`E_UNSUPPORTED_DIR`(不支持方向)。

存储 schema：

```ts
interface HistoryItem {
  id: string; ts: number;
  srcText: string; srcLang: string; tgtLang: string;
  result: string; configSnapshot: TranslateConfig;
}
interface FavoriteItem {
  id: string; ts: number; result: string; note?: string;
}
```

历史落 IndexedDB，收藏落 localStorage（体量小、需即时读取）。

## 6. 边界处理实现要点

| 边界 | 实现策略 |
|---|---|
| 空输入灰态 | `text.trim() === ''` 时提交按钮 disabled，输入框下方提示「请输入内容」 |
| 超长 >2000 | 按 ~1000 字 / 块切分（保留词边界），逐块 `translate` 并显示进度条；合并时按序拼接 |
| 混合语言路由 | §2 步骤 1 按 Unicode 区间逐字路由，英 / 符号原样透传，日文分支走假名规则 |
| 不可译字符 | 原样保留并在前后插入软虚线标记（如 `︵字︶`），`marks[]` 记录坐标供 UI 高亮 |

## 7. 质量护栏落地

准确率测试集 schema（≥2000 条构成）：

```json
{
  "input": "我好喜欢你",
  "expected": "涐恏囍歡伱",
  "ruleTags": ["homophone", "shape"],
  "direction": "han→mars"
}
```

回归脚本伪代码：

```
for case in testSet:
  out = translate(case.input, case.direction, config)
  score = editDistanceApprox(out, case.expected)   // 词级编辑距离近似
  acc += score
  if case.direction == 'mars→han': reverseAcc += (out == case.expected)
regressAll()
repro = reproducibilityCheck()                      // 同输入跑 100 次一致率
if repro < 0.99: BLOCK                              // 可复现率<99% 阻断
if reverseAcc drops > 0.02 vs baseline: BLOCK       // 反向还原下降>2pp 阻断
```

阻断阈值：可复现率 < 99%（绝对硬门槛）、反向还原准确率较基线下降 > 2pp（相对阻断）；词级编辑距离近似准确率的绝对基线（即「规划底线」）由数析在回归基线建立时给定，同样采用「较基线下降 >2pp 即阻断」。规则更新后必须全量回归。

## 8. 推荐技术栈与目录结构

- 框架：React 18 + TypeScript + Vite；状态用 Zustand（响应式配置）。
- 引擎：纯 TS 模块，零运行时依赖，可独立单测。
- 存储：IndexedDB（localForage）+ localStorage。
- 分享卡：Canvas 2D（离屏渲染 → `toBlob` 导出 PNG）。

目录结构：

```
src/
  engine/      # translate 门面 + 流水线 + 可复现 seed
  rules/       # 7 类规则实现
  dictionary/  # 词库 JSON + 加载器 + 繁简映射表
  config/      # 配置 schema + 风格模板 + store
  ui/          # 首页/历史页/设置页 + 组件树
  share/       # canvas 分享卡生成（安全字符优先 + 兜底）
  storage/     # IndexedDB / localStorage 封装
```

组件树：`App → { TranslatePage(汉→火 / 火→汉 Tab) → ResultCard(复制/收藏/分享), HistoryPage, SettingsPage(强度/规则族/繁体/白名单/风格) }`。

## 9. 验收用例（Given/When/Then）

- **R01 汉→火（简）**：Given 配置 homophone+shape 开、强度 mid；When 输入「你好」；Then 输出含火星文候选（如「伱恏」），可复现率 100%。

## 四、并行轨道

- 质量护栏：T05（S1）→ T11（S2）→ 持续周回归，与研发并行，作为 CI 门禁贯穿 S1–S3，规则更新即回归、降 >2pp 阻断。
- 设计：T17 分享卡视觉 S1 即与研发并行定稿供 T14；T06/T16 双入口与设置页视觉随前端同步走查。
- 数据：T03 词库与 T05 测试集 S1 与引擎并行，T18 灰度监控 S3 收口。

## 五、发布门槛与灰度交接

回滚条件（任一触发即回滚并冻结新功能）：翻译成功率 <90% / 可复现率 <99% / 分享卡失败率 >3%。

灰度阶段与退出标准：
- 0→5%（内测）：内部+早鸟邀请码；退出：核心翻译与还原准确率达标、分享卡渲染正常、无 P0 缺陷。
- 5%→30%（分群）：社交娱乐渠道优先；退出：漏斗斜率正向、WSTA 上升、三指标稳定。
- 30%→100%（全量）：退出：成功率≥90%、可复现率≥99%、失败率≤3%，且连续 48h 无回滚触发。
- 灰度期每日回流失败样本 → 规则热更新 → 词库周更。

## 六、风险与缓冲

1. 确定性映射未达 ≥99% 可复现：T02 留 1 SP 收敛；映射确定性优先、禁随机候选、单测锁基线。
2. 简繁误判：T04 Unicode 区间 + 词典优先路由；测试集增简繁混合用例，误判即补规则。
3. 分享卡变形（微信/QQ 过滤）：T17 安全字符优先 + 兜底预览；失败率>3% 回滚降级纯文本卡。
4. 火→汉还原不唯一 / 降 >2pp 阻断：T07+T12 回退「原样保留+标记」；月抽检 ≥200 条人工校准。
5. 前端资源紧张（3–5 人）：T14/T16 设计前置并行，R14 留 1 SP 全栈缓冲；复用组件库，必要时降级风格模板数量。
